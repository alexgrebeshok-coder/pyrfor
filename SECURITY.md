# Security policy

## Supported versions

Security fixes are applied on the `main` branch and released through the normal release process for published packages (see [`CONTRIBUTING.md`](./CONTRIBUTING.md)).

## Reporting a vulnerability

**Please do not open a public GitHub issue for security vulnerabilities.**

Use one of the following:

1. **[GitHub Security Advisories](https://docs.github.com/en/code-security/security-advisories/about-github-security-advisories)** — open a **private** security advisory for this repository when that option is available. Maintainers will coordinate fixes and disclosure there.
2. If private reporting is unavailable for your situation, use GitHub’s **[report abuse / contact support](https://docs.github.com/en/site-policy/github-terms/github-community-guidelines)** paths as documented by GitHub.

Include:

- A short description of the issue and its impact
- Steps to reproduce (proof-of-concept if possible)
- Affected components or packages (for example `@pyrfor/engine`)

We aim to acknowledge reports promptly and work toward a coordinated disclosure timeline.

## After reporting

- You may be asked for follow-up details or to validate a fix before release.
- Credit in release notes or advisories can be arranged if you want it.
