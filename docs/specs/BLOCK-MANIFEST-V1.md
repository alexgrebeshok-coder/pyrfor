# Block Manifest v1

**Spec version:** 1.0.0  
**Status:** Draft — canonical Block SDK target for Pyrfor v1.2  
**Date:** 2026-05-15  
**Depends on:** `PYRFOR-ECOSYSTEM-VISION.md`, `PYRFOR-ECOSYSTEM-STRATEGY.md`

---

## 0. Purpose

`block.json` is the only canonical manifest for a Pyrfor block.

It replaces the old imperative TypeScript `BlockDefinition` interface. A block package is loaded by parsing its manifest, validating the requested permissions/contracts, verifying package metadata, and then executing named lifecycle scripts. The manifest is declarative; it does not embed JavaScript callbacks.

The manifest defines:

- block identity and runtime mode;
- explicit capability tokens;
- contract registry inputs/outputs;
- UI panels and events;
- memory and artifact scopes;
- Optimizer boundaries;
- security, signing, and certification state.

## 1. Non-Goals

| Non-goal | Decision |
|----------|----------|
| Business logic | Lives in block code, not in `block.json` |
| Secret values | Stored in Engine Secrets Manager, never in the manifest |
| Embedded lifecycle callbacks | Forbidden; use named package scripts |
| Untyped cross-block JSON | Forbidden; use Contract Registry references |
| Silent broad permissions | Forbidden; every capability has a reason |

## 2. Package Layout

```text
my-block/
├── block.json
├── package.json
├── contracts/
├── prompts/
├── tools/
├── memory/
├── ui/
├── tests/
├── sbom.cdx.json
└── SIGNATURE.json
```

Rules:

1. `block.json` is required and must be at the package root.
2. `package.json` is required for Node-based blocks; lifecycle scripts referenced by `block.json.scripts` must exist.
3. Directories are optional unless referenced by the manifest.
4. `sbom.cdx.json` is required for `pilot` and `certified` states.
5. `SIGNATURE.json` is generated by publishing/signing tools and is not hand-edited.

## 4. Field Reference

### 4.1 Identity

| Field | Required | Notes |
|-------|----------|-------|
| `$schema` | Recommended | JSON Schema URI |
| `pyrfor_manifest_version` | Yes | Must be `"1"` |
| `id` | Yes | Reverse-DNS identifier, immutable after publication |
| `name` | Yes | Human-readable name |
| `version` | Yes | SemVer block version |
| `description` | Yes | Short description |
| `author` | Yes | Person or organization |
| `license` | Yes | SPDX identifier or `UNLICENSED` |

### 4.2 Runtime

| Field | Required | Allowed values |
|-------|----------|----------------|
| `runtime.mode` | Yes | `trusted-core`, `local-worker`, `wasm`, `container`, `remote` |
| `runtime.engine_version_range` | Yes | SemVer range |
| `runtime.node_version_range` | No | SemVer range |
| `runtime.sandbox` | Yes | `none`, `process-isolated`, `wasm-wasi`, `container-oci` |

`trusted-core` is reserved for first-party Engine components. External blocks should use `local-worker`, `wasm`, or `container`.

### 4.3 Entrypoints

| Field | Required | Notes |
|-------|----------|-------|
| `main` | Yes | Main block module |
| `worker` | No | Background worker |
| `ui` | No | Root UI component |
| `a2a_agent_card` | No | A2A agent card |
| `mcp_server` | No | MCP server launch module |

### 4.4 Lifecycle scripts

Lifecycle:

```text
install -> verify -> migrate -> activate -> run -> suspend -> upgrade -> rollback -> revoke -> uninstall
```

Manifest scripts are command strings. They are not callbacks.

| Script | Required | Trigger |
|--------|----------|---------|
| `install` | No | After package extraction |
| `activate` | No | When enabled in a project |
| `deactivate` | No | When disabled |
| `upgrade` | No | Before new version activation |
| `rollback` | No | After failed upgrade |
| `uninstall` | No | Before removal |
| `test` | Yes | Certification / validation |

Hook environment variables:

| Variable | Meaning |
|----------|---------|
| `PYRFOR_BLOCK_ID` | Block id |
| `PYRFOR_BLOCK_VERSION` | Current block version |
| `PYRFOR_PROJECT_ID` | Active project id, when applicable |
| `PYRFOR_ENGINE_VERSION` | Engine version |
| `PYRFOR_PREV_VERSION` | Previous version during upgrade |
| `PYRFOR_TARGET_VERSION` | Target version during rollback |
| `PYRFOR_DATA_DIR` | Isolated block data directory |

### 4.5 Contracts

All cross-block payloads must use Contract Registry references:

```json
{
  "contracts": {
    "consumes": [
      { "ref": "Document@1", "from": "com.pyrfor.docs-block", "optional": false }
    ],
    "produces": [
      { "ref": "ApprovalEvidence@1" }
    ]
  }
}
```

Contract reference format is `<Name>@<major>`. Patch/minor versions are resolved by the Engine inside the same major line.

### 4.6 Capabilities

Every capability item has:

| Field | Required | Notes |
|-------|----------|-------|
| `token` | Yes | Capability token |
| `reason` | Yes | Human-readable reason visible in Trust Panel |
| `scope` | No | `project`, `block`, `global`, or token-specific scope |
| `expires_after_run` | No | If true, grant expires after one run |

No wildcards are allowed (`fs:*`, `*:write`, etc.).

### 4.7 UI panels

| Field | Required | Allowed values |
|-------|----------|----------------|
| `id` | Yes | Unique within block |
| `slot` | Yes | `left`, `center`, `right`, `bottom`, `modal`, `sidebar` |
| `label` | Yes | UI label |
| `entry` | Yes | Component path |
| `requires_capabilities` | No | Tokens required for panel render |

### 4.8 Events

Events are declarative. Use `<domain>.<entity>.<action>` where possible.

Each emitted event is wrapped by the Engine with `event_id`, `source_block`, `project_id`, `correlation_id`, `causation_id`, `idempotency_key`, `timestamp`, `schema_version`, and `payload`.

### 4.9 Memory scope

| Scope | Meaning |
|-------|---------|
| `project_shared` | Tables visible to other approved blocks in the project |
| `block_private` | Tables only this block can read/write |
| `global_shared` | Engine-wide data; restricted to trusted core |

### 4.10 Artifact types

`artifact_types` declares output types the block can create. Every artifact must be recorded in Artifact Ledger with source refs, model/prompt/block versions, approval state, and lineage id.

### 4.11 Optimizer policy

| Field | Meaning |
|-------|---------|
| `editable` | Whether Optimizer may produce change proposals for this block |
| `editable_fields` | JSON paths Optimizer may propose or auto-apply if auto-apply is explicitly enabled |
| `never_editable` | Fields Optimizer cannot change or propose changing |
| `requires_human_approval` | Fields Optimizer may propose but only Trust Panel can approve |

Optimizer is read-only by default. It can analyze execution traces and propose diffs through Trust Panel. It must not obtain write capability tokens to mutate block packages directly.

Recommended minimum:

```json
{
  "optimizer_policy": {
    "editable": true,
    "editable_fields": ["prompts"],
    "never_editable": ["id", "version", "capabilities", "contracts", "security", "signing"],
    "requires_human_approval": ["runtime", "entrypoints", "scripts", "memory_scope"]
  }
}
```

For critical blocks:

```json
{ "optimizer_policy": { "editable": false } }
```

### 4.12 Security

Security is deny-by-default. Any network, child process, filesystem write, or secret access must be explicit and justified by a capability token.

### 4.13 Signing

`ed25519` is the default open-source package signature for block packages and Tauri updater compatibility. Russian enterprise deployments that require ГОСТ signatures are covered separately by `RU-COMPLIANCE-SCOPE.md`.

### 4.14 Certification

Allowed states:

| State | Meaning |
|-------|---------|
| `dev` | Local development only |
| `internal` | Internal team use |
| `pilot` | Pilot deployment on named projects |
| `certified` | Approved for governed marketplace |
| `revoked` | Blocked by Engine |

Marketplace publication requires at least `pilot`; public marketplace listing requires `certified`.

## 5. Capability Token Model

| Token | Meaning |
|-------|---------|
| `project:read` / `project:write` | Project metadata and WBS |
| `fs:read` / `fs:write` | Filesystem access within approved paths |
| `docs:read` / `docs:write` / `docs:extract` | Document operations |
| `1c:read` / `1c:write` | 1C OData access; write requires approval |
| `memory:read` / `memory:write` | Engine memory, scoped |
| `local-llm:invoke` | Local model calls |
| `cloud-llm:invoke` | Cloud model calls; not allowed in air-gapped mode |
| `model:select` | Runtime model selection |
| `network:outbound` / `network:inbound` | Network access to explicit hosts/ports |
| `mcp:connect` | MCP server connection |
| `a2a:call` | Call another block/agent |
| `event:publish` / `event:subscribe` | Event bus access |
| `artifact:create` / `artifact:read` | Artifact Ledger access |
| `trust-panel:propose` / `trust-panel:notify` | Human approval/notification surface |
| `audit:write` | Audit event emission |

Principles:

1. Least privilege.
2. Every token has a `reason`.
3. No wildcards.
4. Runtime revocation is supported.
5. Token usage is auditable.

## 6. Contract Registry and SemVer

Core contracts:

| Contract | Meaning |
|----------|---------|
| `Document@1` | Source document |
| `EstimateItem@1` | Estimate line item |
| `PaymentCertificate@1` | КС-2/КС-3-like payment certificate |
| `RegulatoryFinding@1` | Clause-level regulatory finding |
| `BIMObject@1` | IFC/BIM object |
| `ScheduleActivity@1` | Schedule activity |
| `SupplyItem@1` | Supply item |
| `InspectionRecord@1` | Field inspection record |
| `ApprovalEvidence@1` | Human approval/review evidence |
| `ChangeOrder@1` | Contract change |

SemVer rules:

| Change | Version impact |
|--------|----------------|
| Add optional field | Patch/minor inside same major |
| Add required field | New minor plus compatibility review |
| Remove/rename/change type | New major |

Blocks reference major versions only (`EstimateItem@1`). The Engine resolves the latest compatible minor/patch.

## 7. Migration Notes

### 7.1 Old `BlockDefinition` mapping

| Old field | New location |
|-----------|--------------|
| `id`, `name`, `version` | top-level fields |
| `engineVersionRange` | `runtime.engine_version_range` |
| `capabilities` | `capabilities[]` token objects |
| `tools` | `entrypoints.main` + `tools/` package contents |
| `skills` | block code/package dependencies; future skills manifest |
| `panels` | `panels[]` |
| `routes` | `entrypoints.mcp_server` or Gateway extension contract |
| `memoryTables` | `memory_scope` + SQL migrations in `memory/` |
| `requiredPermissions` | `capabilities[]` + `security` |
| `onInstall`, `onActivate`, etc. | `scripts.install`, `scripts.activate`, etc. |

### 7.2 Coding Block migration

1. Create `com.pyrfor.coding-block/block.json`.
2. Move install/activation logic into `scripts/on-install.js` and `scripts/on-activate.js`.
3. Declare explicit `fs:read`, `fs:write`, `local-llm:invoke`, `trust-panel:propose`, `memory:*`, and `artifact:*` tokens.
4. Add `optimizer_policy.never_editable` for capabilities, security, signing, contracts.
5. Validate with `pyrfor block validate`.
6. Keep the old wiring behind a feature flag until rollback is proven.

### 7.3 CEOClaw Block migration

1. Create `com.pyrfor.ceoclaw-block/block.json`.
2. Declare consumed contracts from Docs, Estimate, Schedule, Supply Chain, Regulatory blocks.
3. Move `projects`, `risks`, `action_items`, and reports tables into `memory/001_init.sql`.
4. Register Dashboard, Risk, and Timeline panels via `panels[]`.
5. Start as `pilot`, not `certified`.
6. Rollback rule: if manifest activation fails, Engine disables Manifest mode and returns to legacy CEOClaw wiring for the project.

## 9. Acceptance Checklist

### Manifest

- [ ] `block.json` exists at package root.
- [ ] `pyrfor_manifest_version` is `"1"`.
- [ ] `id` is reverse-DNS and immutable.
- [ ] `version` is valid SemVer.
- [ ] `runtime.engine_version_range` is set.
- [ ] All referenced lifecycle scripts exist in `package.json` or package files.
- [ ] `pyrfor block validate` passes.

### Capabilities and security

- [ ] Every token has a non-empty `reason`.
- [ ] No wildcard capability exists.
- [ ] Requested tokens match real code usage.
- [ ] Write tokens require Trust Panel approval where appropriate.
- [ ] `allow_network` and `allow_child_process` are false unless explicitly justified.
- [ ] Memory/CPU limits are set.

### Contracts and artifacts

- [ ] All produced contracts exist in Contract Registry.
- [ ] All consumed contracts are resolvable.
- [ ] Every output artifact type is declared.
- [ ] Artifact Ledger writes include source refs and lineage ids.

### Optimizer

- [ ] `never_editable` includes at least `id`, `version`, `capabilities`, `security`, `signing`.
- [ ] `requires_human_approval` includes at least `runtime`, `entrypoints`, `scripts`.
- [ ] Optimizer never receives direct package-write capability.

### Certification

- [ ] `test` script passes.
- [ ] Acceptance tests cover install, activate, primary run, deactivate.
- [ ] `sbom.cdx.json` exists for `pilot` and `certified`.
- [ ] `SIGNATURE.json` verifies.
- [ ] Certification state matches real review status.

## 10. CLI Commands

```bash
pyrfor block init my-block
pyrfor block validate ./my-block
pyrfor block test ./my-block
pyrfor block pack ./my-block
pyrfor block sign ./my-block --key-id pyrfor-publishing-key-2026
pyrfor block install ./my-block.pyrblock
pyrfor contracts publish contracts/registry/EstimateItem.v1.yaml
```
